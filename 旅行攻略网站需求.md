# 北京旅行攻略网站需求

## 1. 项目目标

开发一个面向移动端的北京旅行攻略网站，聚焦“快速查看北京知名景点、推荐游玩路线、基础出行信息”，整体实现保持轻量，不做复杂社区、登录、支付等能力。

技术实现要求：

- 前端框架：Vue 3
- 构建工具：Vite
- 数据方式：本地 Mock 模拟
- 适配方向：移动端 H5

## 2. 产品定位

该网站面向第一次来北京旅游或短期周末出行用户，提供：

- 北京热门景点浏览
- 景点详情查看
- 推荐一日游 / 两日游路线
- 基础旅行建议与注意事项

核心原则：

- 页面少，结构清晰
- 数据静态可维护
- 视觉简洁，适合手机浏览
- 便于后续从 Mock 平滑切换真实接口

## 3. 用户场景

### 3.1 目标用户

- 计划来北京旅游的游客
- 想快速了解北京经典景点的用户
- 希望参考简单路线安排的移动端用户

### 3.2 典型使用场景

1. 用户打开首页，快速浏览北京热门景点。
2. 用户点击景点卡片，查看景点介绍、开放时间、门票、建议游玩时长等信息。
3. 用户查看推荐路线，决定一日游或两日游行程安排。
4. 用户查看出行建议，了解最佳季节、交通建议和注意事项。

## 4. 功能范围

本期仅做 MVP，包含以下模块：

- 首页
- 景点列表页
- 景点详情页
- 推荐路线页
- 旅行贴士页

不包含的内容：

- 用户注册登录
- 评论、点赞、收藏
- 在线预订门票
- 地图实时导航
- 后台管理系统

## 5. 页面需求

### 5.1 首页

页面目标：快速建立“北京旅行攻略”主题认知，并引导进入核心内容。

页面内容建议：

- 顶部 Banner
  - 标题：北京旅行攻略
  - 副标题：经典景点 / 推荐路线 / 旅行贴士
- 搜索入口
  - 支持按景点名称关键字搜索
  - MVP 可仅做前端本地筛选
- 热门景点区域
  - 展示 4 到 6 个热门景点卡片
- 推荐路线入口
  - 一日游推荐
  - 两日游推荐
- 旅行贴士入口
  - 最佳旅游季节
  - 北京交通建议

交互要求：

- 点击景点卡片进入景点详情页
- 点击路线卡片进入路线详情或路线页
- 点击搜索后筛选景点列表

### 5.2 景点列表页

页面目标：集中浏览北京知名景点。

页面内容建议：

- 顶部标题栏
- 搜索框
- 分类筛选
  - 全部
  - 历史文化
  - 皇家园林
  - 现代地标
  - 胡同街区
- 景点列表卡片

每个景点卡片包含：

- 景点图片
- 景点名称
- 简短简介
- 标签
- 参考门票
- 建议游玩时长

交互要求：

- 支持关键字搜索
- 支持分类切换筛选
- 点击卡片进入详情页

### 5.3 景点详情页

页面目标：展示单个景点的完整攻略信息。

页面内容建议：

- 景点主图
- 景点名称
- 景点评分（可用模拟数据）
- 景点标签
- 景点简介
- 基础信息区
  - 开放时间
  - 门票价格
  - 建议游玩时长
  - 建议游玩季节
  - 地址
- 攻略说明
  - 必看亮点
  - 游玩建议
  - 注意事项
- 相关推荐
  - 2 到 3 个关联景点

建议景点范围：

- 故宫博物院
- 天安门广场
- 八达岭长城
- 颐和园
- 天坛公园
- 南锣鼓巷
- 什刹海
- 鸟巢 / 水立方

### 5.4 推荐路线页

页面目标：给用户简单直接的出行安排参考。

页面内容建议：

- 一日游路线
  - 例如：天安门广场 -> 故宫 -> 景山公园 -> 南锣鼓巷
- 两日游路线
  - 例如：
    - Day1：天安门广场 -> 故宫 -> 天坛
    - Day2：颐和园 -> 圆明园 -> 鸟巢
- 每条路线包含：
  - 路线名称
  - 路线简介
  - 行程节点
  - 建议出发时间
  - 适合人群

交互要求：

- 点击路线卡片查看路线详情
- 路线节点可跳转到景点详情页

### 5.5 旅行贴士页

页面目标：补充用户决策需要的基础信息。

页面内容建议：

- 最佳旅行季节
- 穿衣建议
- 北京交通方式说明
  - 地铁
  - 公交
  - 打车
- 美食建议
  - 北京烤鸭
  - 炸酱面
  - 豆汁等可做简单介绍
- 注意事项
  - 旺季预约
  - 身份证/证件携带
  - 景区安检

## 6. 信息架构

建议路由结构：

```text
/                    首页
/spots               景点列表页
/spots/:id           景点详情页
/routes              推荐路线页
/routes/:id          路线详情页（可选）
/tips                旅行贴士页
```

底部导航建议：

- 首页
- 景点
- 路线
- 贴士

## 7. 视觉与交互要求

### 7.1 设计风格

- 整体风格清爽、简洁、偏旅游内容展示
- 色彩建议以红色、金色、米白色为主，呼应北京文化氛围
- 卡片式布局，便于移动端阅读

### 7.2 移动端要求

- 设计宽度按 `375px` 参考
- 兼容主流移动端屏幕
- 列表区域支持自然纵向滚动
- 触控点击区域不小于 `44px`

### 7.3 交互细节

- 页面切换有基础过渡即可，不需要复杂动画
- 图片建议使用圆角卡片样式
- 加载状态可使用骨架屏或简单 loading
- 空数据时显示占位提示

### 8.3 API 模拟封装建议

在 `src/api/travel.js` 中统一封装数据获取方法，模拟真实接口：

```js
import spots from "../mock/spots";
import routes from "../mock/routes";
import tips from "../mock/tips";
import banners from "../mock/banners";

const wait = (time = 300) => new Promise((resolve) => setTimeout(resolve, time));

export async function getBanners() {
  await wait();
  return banners;
}

export async function getSpots(params = {}) {
  await wait();
  const { keyword = "", category = "全部" } = params;
  return spots.filter((item) => {
    const matchKeyword = item.name.includes(keyword);
    const matchCategory = category === "全部" || item.category === category;
    return matchKeyword && matchCategory;
  });
}

export async function getSpotDetail(id) {
  await wait();
  return spots.find((item) => item.id === Number(id));
}

export async function getRoutes() {
  await wait();
  return routes;
}

export async function getRouteDetail(id) {
  await wait();
  return routes.find((item) => item.id === Number(id));
}

export async function getTips() {
  await wait();
  return tips;
}
```

说明：

- 使用异步 `Promise + setTimeout` 模拟网络延迟
- 后续接入真实后端时，仅需替换 `api/travel.js` 内部实现
